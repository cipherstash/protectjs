## SUGGESTIONS (Would Improve Quality)

**SUG-1: Missing error recovery strategy for database connection failures:**
- Description: Plan lacks explicit error handling for database connection failures during test setup. Task 5 Step 1 mentions "Ensure database is available" but doesn't specify what to do if it's not, or how to provide actionable feedback to developers.
- Location: Task 8, Step 1 ("Ensure database is available")
- Benefit: Prevents cryptic test failures when DATABASE_URL is misconfigured or database is unreachable. Developers would get clear guidance on what went wrong.
- Action: Add explicit verification step with clear error messages. For example:
  ```typescript
  // In beforeAll, add connection verification:
  try {
    await client`SELECT 1`
  } catch (error) {
    throw new Error(
      `Failed to connect to database. Check DATABASE_URL is set and database is accessible.\n` +
      `Error: ${error instanceof Error ? error.message : String(error)}`
    )
  }
  ```

**SUG-2: Test cleanup strategy unclear for partial failures:**
- Description: Task 5 uses `afterAll` to clean up seeded data by ID array. If `beforeAll` fails partway through seeding (e.g., after inserting 8 of 15 records), `seedDataIds` will be incomplete and cleanup will be partial, leaving orphaned test data.
- Location: Task 5, `afterAll` cleanup in docs.test.ts (lines 600-605)
- Benefit: Guarantees clean test state even when setup fails partially. Prevents accumulation of test data over multiple runs.
- Action: Use a more robust cleanup strategy:
  ```typescript
  afterAll(async () => {
    // Clean up by timestamp range instead of ID array
    await db.delete(transactions)
      .where(
        and(
          gte(transactions.createdAt, testStartTime),
          lte(transactions.createdAt, testEndTime)
        )
      )
    await client.end()
  }, 30000)
  ```
  Or track IDs incrementally during insertion rather than relying on a single bulk operation.

**SUG-3: Seed data dates hardcoded to 2024-12:**
- Description: Fixture uses hardcoded December 2024 dates. If documentation examples rely on "recent" data or relative date ranges, tests could fail when run in future years.
- Location: Task 4, doc-seed-data.ts (lines 398-484)
- Benefit: Tests remain valid long-term without requiring manual updates to seed data dates.
- Action: Use relative dates from current time:
  ```typescript
  const now = Date.now()
  const dayMs = 24 * 60 * 60 * 1000
  // ... then use now - (5 * dayMs) for "5 days ago", etc.
  ```
  Or document the assumption explicitly with a comment explaining why December 2024 was chosen.

**SUG-4: No validation that documentation files exist before running tests:**
- Description: Task 5 gracefully skips tests if docs don't exist (lines 616-622, 673-679), but this could mask a real problem (file moved, renamed, or deleted). CI would pass with all tests skipped.
- Location: Task 5, docs.test.ts try/catch blocks for reading markdown
- Benefit: Prevents false positives in CI when documentation files are accidentally removed or moved.
- Action: Add explicit validation:
  ```typescript
  // In a separate test or in beforeAll:
  it('documentation files exist', () => {
    expect(() => readFileSync(docsPath, 'utf-8')).not.toThrow()
  })
  ```
  This way, if files are missing, the test suite fails explicitly rather than silently skipping all tests.

**SUG-5: Task 2 test coverage incomplete for edge cases:**
- Description: markdown-parser tests cover basic cases but miss important edge cases: consecutive code blocks without headers, nested code blocks (triple backticks inside a block), empty code blocks, blocks at start/end of file, malformed blocks (missing closing ```).
- Location: Task 2, markdown-parser.test.ts (lines 60-129)
- Benefit: Parser is more robust against real-world markdown variations, reducing false negatives and runtime failures.
- Action: Add edge case tests:
  - Empty ts:run block: `\`\`\`ts:run\n\`\`\``
  - Block at line 1 of file (lineNumber should be 2)
  - Two consecutive ts:run blocks without intervening text
  - Malformed block (no closing ```) - should be skipped or handled gracefully

**SUG-6: No verification that extracted code blocks are syntactically valid TypeScript:**
- Description: markdown-parser extracts code as strings but doesn't validate syntax. A typo in documentation (missing semicolon, unclosed brace) will only be caught at execution time in Task 5, making debugging harder.
- Location: Task 2, markdown-parser.ts implementation
- Benefit: Earlier detection of documentation syntax errors, clearer error messages pointing to specific line numbers in markdown.
- Action: Add optional syntax validation in extractExecutableBlocks:
  ```typescript
  import { createSourceFile, ScriptTarget } from 'typescript'

  // After extracting code:
  const sourceFile = createSourceFile(
    'temp.ts',
    block.code,
    ScriptTarget.Latest,
    true
  )
  if (sourceFile.parseDiagnostics.length > 0) {
    console.warn(`Syntax error in ${section} at line ${lineNumber}`)
  }
  ```

**SUG-7: code-executor uses Function() constructor without security note:**

- Location: Task 3, code-executor.ts line 340

**SUG-13: No handling for documentation code blocks that intentionally demonstrate errors:**
