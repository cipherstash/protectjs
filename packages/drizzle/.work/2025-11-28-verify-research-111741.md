## Key Findings

### Finding 1: Changeset Automation Updated peerDependencies to Unpublished Versions
- **Source:** Git commit `a6d0fbc` (Version Packages), file: `packages/drizzle/package.json`
- **Evidence:**
  ```diff
  -"@cipherstash/protect": ">=10",
  -"@cipherstash/schema": ">=2",
  +"@cipherstash/protect": ">=10.1.1",
  +"@cipherstash/schema": ">=2.0.1",
  ```
- **Confidence:** HIGH
- **Implication:** The changeset automation (via GitHub Actions) bumped the peerDependency version requirements as part of the "Version Packages" commit. This is expected behavior when dependencies are updated via changesets.

### Finding 2: The Bumped Versions Don't Exist on npm Registry
- **Source:** npm registry query via `npm view @cipherstash/protect versions` and `npm view @cipherstash/schema versions`
- **Evidence:**
  - Latest published `@cipherstash/protect` on npm: **10.1.0**
  - Latest published `@cipherstash/schema` on npm: **2.0.0**
  - Required by drizzle after changeset: `>=10.1.1` and `>=2.0.1`
- **Confidence:** HIGH
- **Implication:** The versions 10.1.1 and 2.0.1 referenced in peerDependencies only exist in the workspace (monorepo), not on the npm registry. This creates a mismatch between package.json requirements and what's available publicly.

### Finding 3: Workspace Versions Exist But Aren't Published
- **Source:** Files `packages/protect/package.json` and `packages/schema/package.json` in workspace
- **Evidence:**
  - Workspace version of `@cipherstash/protect`: **10.1.1**
  - Workspace version of `@cipherstash/schema`: **2.0.1**
  - These versions were created by the changeset automation but not yet published to npm
- **Confidence:** HIGH
- **Implication:** The changeset automation created the version bump commits but the publishing step (which should happen in the release workflow) has not yet occurred. This is a timing issue in the release process.

### Finding 4: The Lockfile Was Never Updated After peerDependency Changes
- **Source:** Git commit `a690dde` (merge of PR #224), file: `pnpm-lock.yaml`
- **Evidence:**
  - After commit `a6d0fbc`, the lockfile still references `@cipherstash/protect` with `specifier: '>=10'` and `version: 10.0.1`
  - The merge commit `a690dde` shows no diff in pnpm-lock.yaml (verified via `git diff a6d0fbc a690dde -- pnpm-lock.yaml`)
  - Current main branch lockfile still shows the old specifier
- **Confidence:** HIGH
- **Implication:** The changeset automation updated package.json but did NOT run `pnpm install` to update the lockfile. This left the lockfile in an inconsistent state with package.json.

### Finding 5: The Build Error Indicates Frozen Lockfile Validation
- **Source:** Task description (CI logs)
- **Evidence:**
  ```
  ERR_PNPM_OUTDATED_LOCKFILE  Cannot install with "frozen-lockfile" because
  pnpm-lock.yaml is not up to date with packages/drizzle/package.json

  Failure reason:
  specifiers in the lockfile don't match specifiers in package.json:
  * 1 dependencies are mismatched:
    - @cipherstash/protect (lockfile: >=10, manifest: >=10.1.1)
  ```
- **Confidence:** HIGH
- **Implication:** Some CI environment (likely a developer's local build or a separate CI job not shown in .github/workflows) is running `pnpm install --frozen-lockfile`, which validates lockfile consistency. The test.yml and release.yml workflows use plain `pnpm install` without the frozen flag.

### Finding 6: The Proposed Fix Adds workspace:* to devDependencies
- **Source:** Git commit `8d04442`, file: `packages/drizzle/package.json`
- **Evidence:**
  ```diff
  "devDependencies": {
  +  "@cipherstash/protect": "workspace:*",
     "@cipherstash/schema": "workspace:*",
  ```
- **Confidence:** HIGH
- **Implication:** By adding `@cipherstash/protect` as a dev dependency with `workspace:*` protocol, pnpm will resolve it from the local workspace instead of trying to fetch the unpublished version from npm. This matches the existing pattern for `@cipherstash/schema`.

### Finding 7: The Fix Follows Existing Monorepo Patterns
- **Source:** Survey of other packages in monorepo
- **Evidence:**
  - `packages/drizzle/package.json` already had `"@cipherstash/schema": "workspace:*"` in devDependencies
  - `packages/protect-dynamodb/package.json` has `"@cipherstash/protect": "workspace:*"` in BOTH devDependencies AND peerDependencies (unusual pattern)
  - `packages/protect/package.json` has `"@cipherstash/schema": "workspace:*"` in devDependencies
- **Confidence:** HIGH
- **Implication:** Using `workspace:*` in devDependencies for workspace packages is an established pattern in this monorepo. The fix is consistent with existing practices.

### Finding 8: The Lockfile Changes Show Workspace Resolution
- **Source:** Git diff of commit `8d04442`, file: `pnpm-lock.yaml`
- **Evidence:**
  ```diff
  packages/drizzle:
    dependencies:
  -   '@cipherstash/protect':
  -     specifier: '>=10'
  -     version: 10.0.1
    devDependencies:
  +   '@cipherstash/protect':
  +     specifier: workspace:*
  +     version: link:../protect
  ```
  - Also removed multiple old protect-ffi platform-specific packages from lockfile (0.17.1 versions)
  - Retained only the current 0.18.1 versions
- **Confidence:** HIGH
- **Implication:** The fix correctly moves protect from dependencies to devDependencies and uses workspace linking. The lockfile cleanup is a side effect of pnpm resolving the dependency tree more efficiently.

## Patterns Observed

### Pattern 1: Changeset Automation Timing Issue
The changeset automation updates package.json files (including peerDependencies) and creates version commits, but doesn't ensure that:
1. The new versions are immediately published to npm
2. The lockfile is updated to reflect changes

This creates a window where the workspace is in an inconsistent state.

### Pattern 2: peerDependencies vs devDependencies Resolution
In a monorepo with workspace packages:
- **peerDependencies** with version ranges (e.g., `>=10.1.1`) cause pnpm to look for that version on npm
- **devDependencies** with `workspace:*` cause pnpm to resolve from the local workspace
- When a peerDependency version doesn't exist on npm yet, adding it to devDependencies as `workspace:*` allows local development to continue

### Pattern 3: Existing Inconsistency in protect-dynamodb
The `@cipherstash/protect-dynamodb` package has an unusual pattern:
```json
{
  "peerDependencies": {
    "@cipherstash/protect": "workspace:*"
  },
  "devDependencies": {
    "@cipherstash/protect": "workspace:*"
  }
}
```
Using `workspace:*` in peerDependencies is non-standard and may cause issues when the package is published to npm.

## Root Cause Analysis

### Primary Root Cause
**Changesets bumped peerDependency version requirements to versions that don't exist on npm (10.1.1, 2.0.1), creating a lockfile/manifest mismatch.**

### Contributing Factors
1. **Timing gap between version bump and publish:** The "Version Packages" commit (a6d0fbc) created new versions locally but they weren't published to npm registry
2. **Lockfile not updated:** The changeset automation or subsequent merge didn't run `pnpm install` to update the lockfile to match the new peerDependency specifiers
3. **peerDependencies resolution behavior:** pnpm tries to resolve peerDependencies from npm, not from workspace, causing the version mismatch

### Why This Is Accurate
- **Evidence-based:** Multiple sources confirm each step of the failure chain
- **Timeline verified:** Git history shows exact sequence of changes
- **Npm registry verified:** Confirmed missing versions via npm view command
- **Lockfile state confirmed:** Git diffs show lockfile was never updated after package.json changes

## Fix Assessment

### What the Fix Does
The fix (commit 8d04442) adds `"@cipherstash/protect": "workspace:*"` to the drizzle package's devDependencies, which:
1. Allows pnpm to resolve `@cipherstash/protect` from the local workspace instead of npm
2. Updates the lockfile to reference the workspace version via symlink
3. Removes the dependency entry that was trying to resolve from npm

### Is the Fix Appropriate?
**Yes, with caveats:**

**Strengths:**
- ✅ Follows existing pattern (schema was already handled this way)
- ✅ Allows immediate local development without waiting for npm publish
- ✅ Resolves the lockfile mismatch error
- ✅ Minimal change, low risk
- ✅ Consistent with monorepo best practices

**Potential Issues:**
- ⚠️ Doesn't address the root cause (unpublished versions on npm)
- ⚠️ Could hide the symptom if packages are never published
- ⚠️ The protect-dynamodb package has workspace:* in peerDependencies which is unusual and may indicate a different approach was considered

### Is the Fix Complete?
**Yes, for immediate resolution. No, for long-term health.**

**Immediate:** The fix resolves the build failure and allows development to continue.

**Long-term considerations:**
1. Versions 10.1.1 and 2.0.1 should still be published to npm
2. The changeset/release process should be reviewed to prevent this timing issue
3. Consider whether lockfile should be updated automatically by changeset automation
4. Document the pattern of using workspace:* for workspace packages in devDependencies

## Summary

The root cause analysis is **accurate and well-founded**. The changesets automation bumped peerDependency version requirements to `>=10.1.1` and `>=2.0.1`, but these versions only exist in the workspace and haven't been published to npm. This created a mismatch between package.json and pnpm-lock.yaml that caused frozen-lockfile validation to fail.

The proposed fix is **appropriate and follows existing patterns** in the monorepo. Adding `@cipherstash/protect` to devDependencies with `workspace:*` allows pnpm to resolve from the workspace, matching the existing pattern for `@cipherstash/schema`.

The fix is **complete for immediate purposes** but doesn't address the underlying timing issue in the release process. The unpublished versions should still be published to npm, and the release workflow should be reviewed to prevent this scenario in the future.

**Final Assessment: APPROVE THE FIX with follow-up actions to publish the pending versions and review the release process.**
